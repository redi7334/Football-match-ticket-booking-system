1. Introduction

This report documents the design, implementation, and deployment of a web-based Football Match Ticket Booking System developed as a course project for Programming II. The application allows football fans to register, browse upcoming matches, and book tickets online, while giving administrators full control over teams, fixtures, customers, and bookings.

The project was chosen because online ticketing is a familiar, real-world domain that combines several core concepts of object-oriented programming: encapsulation of state and behavior in dedicated classes, clear separation between data and logic, association between objects (a Ticket links a Customer with a Match), and a service layer that coordinates them. It also offered an opportunity to take the assignment one step further by deploying the result on the public internet rather than only running it locally.

The system is written entirely in Java using only the standard JDK, with no external libraries or frameworks. The user interface is delivered as HTML pages served by an embedded HTTP server included in the JDK. The result is a complete, self-contained web application that can be compiled with a single command and accessed from any browser.

2. Problem Statement and Requirements Analysis

The problem addressed is the management of football-match ticket sales for a small club or league office. Without a dedicated system, the office relies on spreadsheets and manual lists to track which customer has booked which seat, leading to double-bookings, lost data, and slow service. A simple web application removes these issues by storing all data in one place and exposing clear, role-restricted operations.

2.1 Functional requirements

  • Manage customers: register, edit, and delete customer profiles (admin), or self-register as a client.
  • Manage teams: register and edit football teams (admin only).
  • Manage matches: schedule new fixtures, edit details such as date, stadium, price, and seat capacity, and delete matches (admin only).
  • Browse matches: any signed-in user can see upcoming and past fixtures with availability indicators.
  • Book tickets: clients book a single ticket for themselves; admins can book on behalf of any customer.
  • Cancel tickets: clients can cancel their own tickets; admins can cancel any ticket.
  • Authentication and authorization: every action requires a login; clients and admins see different menus and have different permissions.
  • Reports: admins see live counts of customers, teams, matches, tickets, and total revenue.

2.2 Non-functional requirements

  • Implemented in pure Java with no external libraries, so the entire project compiles with a single javac command.
  • Accessible from any modern browser; no client installation required.
  • Passwords stored hashed (never in plaintext) and sessions managed through HTTP-only cookies.
  • Deployable to a public URL through a Docker container and a free hosting platform.
  • Code organized in small, single-responsibility classes to support a clean, maintainable design.

2.3 Actors

Two roles interact with the system. The administrator manages all data and oversees bookings. The client is a regular user who self-registers, browses matches, and books tickets for personal use. The system never allows a client to view other clients' data or to perform admin operations.

3. System Design

The system is built around object-oriented principles. Each entity in the domain has its own class that encapsulates its data and behavior, while a central service object coordinates the relationships between them. The web layer talks to the model only through service methods; it never reaches inside the objects directly.

3.1 Class structure

The application is composed of ten Java classes grouped into three layers. The model layer represents the business entities. The service layer holds the data and exposes operations on it. The presentation layer translates HTTP requests into service calls and renders HTML.

  • Customer — a person who can book tickets. Stores name, email, phone, and a unique ID.
  • Team — a football club with a name and home city.
  • Match — a scheduled fixture between two teams. Owns the ticket price and the available-seat count, with methods to reserve and release seats.
  • Ticket — a booked seat that links a Customer to a Match, with a unique seat number and a booking timestamp.
  • User — a login account with a role (ADMIN or CLIENT). Client users are linked to a Customer profile.
  • BookingSystem — the central service. Holds the lists of customers, teams, matches, and tickets, and exposes register/find/update/remove and ticket-booking operations.
  • AuthService — manages users and login sessions, hashes passwords, and links new clients to a Customer record.
  • WebServer — the HTTP layer. Defines all routes, authenticates requests, enforces role restrictions, and renders pages.
  • Html — helpers for HTML escaping, page layout, role-aware navigation, and inline CSS.
  • Main — the entry point. Builds the system, seeds the default admin, and starts the web server.

3.2 Object relationships

A Match owns references to two Team objects (home and away). A Ticket owns references to one Match and one Customer, capturing the seat number and price at the moment of booking. A client User references the ID of its linked Customer. The BookingSystem owns the master lists and is the single source of truth for the application state. AuthService owns the list of Users and the active sessions; it calls into BookingSystem when a self-registering client needs a Customer record created.

4. Implementation

The application uses only the standard Java Development Kit, version 17. No build tool (Maven, Gradle) and no third-party libraries are required. All ten source files compile into bytecode through a single javac *.java command, and the program runs with java Main.

4.1 Web layer

The HTTP server is the com.sun.net.httpserver.HttpServer class included in the JDK. The WebServer class registers a handler for each top-level path (/login, /matches, /tickets, etc.) and dispatches requests by method (GET or POST). HTML responses are built as plain Strings and sent back with the appropriate content type. After every successful POST the server replies with an HTTP 303 redirect, which prevents accidental double-submissions when the user refreshes the page.

All visible text rendered into HTML is passed through Html.esc, an escaping function that replaces special characters with their HTML entities. This eliminates the risk of cross-site scripting from user-supplied values such as names, team names, and stadium names.

4.2 Data persistence

All state lives in memory inside the BookingSystem and AuthService instances, stored in ArrayList collections. This decision keeps the project simple and dependency-free; the trade-off is that data is reset on every server restart. Sample teams, matches, and customers are loaded automatically at startup so the application is immediately usable.

4.3 User interface

Pages share a common layout produced by Html.page. The header uses a navy-to-teal gradient with a custom SVG logo. Navigation changes by role: an unauthenticated visitor sees only Login and Register; a client sees Home, Matches, Book ticket, My tickets, and My profile; an administrator sees the full menu plus Customers, Teams, Tickets, Users, and Reports. Match listings are rendered as ticket-style cards with rounded corners and an availability badge (Available, Few seats left, or Sold out), giving the interface a clear thematic identity.

5. Authentication and Authorization

Authentication is implemented entirely inside AuthService. Passwords are stored as salted SHA-256 hashes: each user has a 16-byte random salt generated at registration, and the stored value is the hex-encoded digest of salt concatenated with the plaintext password. Plaintext passwords are never written to memory beyond the immediate verification step.

After a successful login, the server generates a 32-byte random session identifier and stores the mapping from session ID to user ID in a ConcurrentHashMap. The session ID is sent back to the browser as an HTTP cookie named MTSESSION, marked HttpOnly and SameSite=Lax, with a 24-hour lifetime. On every subsequent request the WebServer reads the cookie, looks up the user, and either serves the page, redirects to the login form, or returns an HTTP 403 Forbidden response if the role does not permit the action.

6. Deployment

A small Dockerfile based on the official Eclipse Temurin Java 17 image copies the source files, compiles them with javac, and runs java Main. Cloud platforms such as Render set a PORT environment variable that the application reads at startup, falling back to 8080 for local development. The repository is hosted on GitHub; pushing a new commit triggers an automatic rebuild and redeploy on Render. The same Dockerfile works without modification on other Java-friendly platforms such as Railway, Fly.io, and Google Cloud Run.

8. Conclusion

The Football Match Ticket Booking System satisfies the goals set out at the beginning of the project. It demonstrates the principles of object-oriented programming through a clean separation of model, service, and presentation layers, with each responsibility encapsulated in its own class. It also goes beyond the minimum requirements by offering a real graphical interface served over HTTP, role-based authentication, and deployment to a public URL.

Possible improvements for future versions include persisting data to a database (for example SQLite or PostgreSQL) so that bookings survive a restart, sending email confirmations after a successful booking, and adding payment integration through a service such as Stripe. The current architecture, with its clearly separated layers, would accommodate these additions without significant refactoring.
